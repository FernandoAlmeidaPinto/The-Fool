## Overview

Admin functionality to create and manage tarot/oracle decks with cards (image upload to MinIO). Public browsing for all authenticated users — list decks, view card grid, view card detail. Cards are subdocuments within the deck document (1:1 relationship, no separate collection).

## 1. Data Model

### Deck (collection `decks`)

```
Deck {
  _id: ObjectId
  name: string              // "Rider-Waite Tarot"
  description: string
  type: string              // enum: "tarot", "lenormand", "oracle"
  coverImage: string | null // URL in MinIO (optional deck cover)
  cards: Card[]             // subdocument array
  createdAt: Date
  updatedAt: Date
}
```

### Card (subdocument within Deck)

```
Card {
  _id: ObjectId             // auto-generated by Mongoose for subdocs
  title: string             // "The Fool", "A Imperatriz"
  description: string       // Card meaning/description
  image: string             // Image URL in MinIO
  order: number             // Position in deck for sorting
}
```

Cards are subdocuments — a single query fetches the deck with all its cards. The `order` field supports future reordering (drag-and-drop UI is out of scope; new cards receive the next sequential number).

### Deck Type Enum

```typescript
export const DECK_TYPES = {
  TAROT: "tarot",
  LENORMAND: "lenormand",
  ORACLE: "oracle",
} as const;

export type DeckType = (typeof DECK_TYPES)[keyof typeof DECK_TYPES];
```

## 2. MinIO (S3-Compatible Storage)

### Docker Compose

Add MinIO service to existing `docker-compose.yml`:

```yaml
minio:
  image: minio/minio
  ports:
    - "9000:9000"    # S3-compatible API
    - "9001:9001"    # Web console
  environment:
    MINIO_ROOT_USER: minioadmin
    MINIO_ROOT_PASSWORD: minioadmin
  volumes:
    - minio_data:/data
  command: server /data --console-address ":9001"
```

### Environment Variables

Added to `.env.example`:

```
S3_ENDPOINT=http://localhost:9000
S3_ACCESS_KEY=minioadmin
S3_SECRET_KEY=minioadmin
S3_BUCKET=the-fool-cards
S3_REGION=us-east-1
```

### Storage Library

`lib/storage/s3.ts` — wrapper using `@aws-sdk/client-s3`. Works with MinIO, S3, and R2 by changing env vars only. Provides:

- `uploadFile(file: Buffer, key: string, contentType: string): Promise<string>` — uploads and returns the public URL
- `deleteFile(key: string): Promise<void>` — deletes a file
- `getPublicUrl(key: string): string` — constructs the public URL for a key

Bucket is created automatically on first upload if it doesn't exist (via `CreateBucketCommand` with error handling for "bucket already exists").

Images are served via the public URL of MinIO in development. The bucket policy is set to public-read for the cards bucket.

## 4. Admin Pages — Deck & Card CRUD

### Routes

- `/admin/decks` — list all decks (table: name, type badge, card count, actions)
- `/admin/decks/new` — create deck form (name, description, type select)
- `/admin/decks/[id]/edit` — edit deck details + card list with "Adicionar Carta" button
- `/admin/decks/[id]/cards/new` — add card form (title, description, image upload)
- `/admin/decks/[id]/cards/[cardId]/edit` — edit card (title, description, replace image)

### Permission

All admin deck pages check `admin:decks` permission.

### Card Upload Flow

1. Admin fills title + description, selects image file via `<input type="file">`
2. Server Action receives FormData, extracts file
3. Uploads to MinIO via `lib/storage/s3.ts` with key `decks/{deckId}/{cardId}.{ext}`
4. Saves the image URL in the card subdocument
5. Redirects back to deck edit page

### File Structure (admin)

```
lib/
  storage/
    s3.ts                          # S3/MinIO client wrapper
  decks/
    model.ts                       # Deck + Card schemas
    constants.ts                   # DECK_TYPES enum
    service.ts                     # CRUD operations
app/(dashboard)/admin/
  decks/
    page.tsx                       # List decks
    actions.ts                     # Server Actions (deck + card CRUD)
    new/page.tsx                   # Create deck
    [id]/edit/page.tsx             # Edit deck + card list
    [id]/cards/
      new/page.tsx                 # Add card with image upload
      [cardId]/edit/page.tsx       # Edit card
```

## 5. Public Visualization

### Sidebar

New item "Baralhos" in the main nav section of the sidebar (between "Cursos" and the "Conta" section). No permission required — all authenticated users see it.

### Routes

- `/baralhos` — list available decks (cards showing cover image or first card image, name, type badge, card count)
- `/baralhos/[id]` — card grid for a deck (responsive: 2 cols mobile, 3 tablet, 4 desktop; card image in ~2:3 aspect ratio + title below)
- `/baralhos/[id]/carta/[cardId]` — card detail (large centered image, title, description, prev/next navigation between cards of the same deck)

### File Structure (public)

```
app/(dashboard)/
  baralhos/
    page.tsx                       # Deck list
    [id]/
      page.tsx                     # Card grid
      carta/
        [cardId]/page.tsx          # Card detail
```

### Page Titles

Added to the header's page title mapping:

```typescript
"/baralhos": "Baralhos",
// Dynamic: /baralhos/[id] → deck name
// Dynamic: /baralhos/[id]/carta/[cardId] → card title
```

For dynamic routes (deck name, card title), the page components set the title via document metadata or a client-side title component.
